The inspiration · how this differs

Optimized for adoption, not autonomy

Traditional AI factories ask: how much can run without a human? We ask a different question: how well does it fit the humans — and the organization — you already have?

  • Built for the enterprise — designed to adopt into an existing org's teams, conventions, and compliance, not to replace them.
  • Augment roles; don't automate them away — the goal isn't a removed human, it's an amplified one. Human-in-the-loop is a design primitive, not a safety setting.
  • A platform, not a prescription — most factories ship one opinionated workflow and ask you to conform. This ships an engine and a vocabulary; domain-specific solutions are options to adopt, fork, or ignore.
  • Inclusive of the whole company — engineering is one tenant. Anything with inputs, work, and quality bars — design, requirements, marketing, ops — is the same primitives.
  • It doesn't prescribe the steps — the flow is data. There is no built-in “right way,” and that is precisely what makes broad adoption possible.

“Specs are data; the engine is the only code” is an adoption strategy, not just an architecture — a non-prescriptive, forkable substrate is what lets diverse teams make the factory their own.

How is this possible?

A factory's behavior is captured in a spec

  • Workers — the units of work (architect, coder…)
  • Signature — typed consumes / produces
  • Edges — typed Artifacts wiring Workers into a DAG
  • Checks — the gates output must pass
  • Adapter — the brain, an LLM from a catalog
  • Resources — the hands (GitHub · Figma · AWS)

The engine just load · resolve · run · check · fold — it never branches on which role it is running.

What a spec actually looks like

The whole definition of a role — no code

# workers/coder.api.worker.yaml
kind: Worker
id:    coder.api            # its "role" is just its identity
skill: implement-endpoint   # the unit of work — a repo skill
tier:  deep-reasoning       # capability alias → catalog picks the model

consumes:                      # typed inputs
  - { name: contract,  schema: openapi.v1 }
  - { name: migration, schema: sql-migration.v1 }
produces:                      # typed outputs
  - { name: handlers,  schema: source.v1, cardinality: many }

resources:   [ github ]                         # its hands
permissions: { access: read-write, scope: "api/**" } # least-privilege
checks:      [ typecheck, lint, unit, judge.api-conventions ]

Nothing compiles alongside this file. The same engine that runs it runs every other role — by loading a different spec.
